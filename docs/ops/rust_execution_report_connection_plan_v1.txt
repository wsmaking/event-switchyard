Rust ExecutionReport Mainline Connection Plan (Step3 start)

Objective
- Connect exchange ExecutionReport flow to Rust order state update path so that state changes are reflected consistently in store, audit, bus, and SSE.

Current gap
- `gateway-rust/src/engine/exchange_worker.rs` receives reports in callback but only logs.
- `ShardedOrderStore::apply_execution_report` exists but is not wired from exchange callback.
- `OrderUpdated` style fan-out (Audit/Bus/SSE) is partially present on order accept/cancel paths only.

Target flow
1. ExecutionReport ingress
- In exchange callback, map exchange report to internal `store::ExecReport`.
- Resolve `account_id` for the order using `OrderIdMap` (or fallback lookup).

2. State apply
- Call `sharded_store.apply_execution_report(&exec_report, account_id)`.
- Preserve existing invariants:
  - terminal status is sticky
  - filled quantity is monotonic (`max(prev, reported)`)
  - reject after any fill does not downgrade to REJECTED

3. Side effects on effective update
- Emit Audit event (`ExecutionReportApplied` or `OrderUpdated`) with before/after fields.
- Publish Bus event with normalized payload (orderId/accountId/status/filledQty/reason).
- Publish SSE events to both order stream and account stream.

4. Error handling / retries
- If account resolution fails, count metric and skip apply (avoid panic in hot path).
- If Audit/Bus publish fails, keep store state authoritative and increment failure counters.

5. Observability
- Add counters/histograms:
  - `gateway_exec_report_applied_total`
  - `gateway_exec_report_dropped_total`
  - `gateway_exec_report_fanout_error_total`
  - apply latency histogram

6. Rollout strategy
- Phase A: wire update path behind env flag `RUST_EXEC_REPORT_APPLY=1`.
- Phase B: enable fan-out behind `RUST_EXEC_REPORT_FANOUT=1`.
- Phase C: default-on after soak and contract/integration tests pass.

Test plan
- Unit tests
  - exchange status -> store status mapping
  - account resolution fallback behavior
  - fan-out payload content
- Contract tests (already added in Step2)
  - risk decision fixture
  - reject reason fixture
  - state transition fixture
- Integration test
  - feed ExecutionReport and assert store/audit/bus/sse updates are consistent.
